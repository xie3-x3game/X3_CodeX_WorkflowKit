# Dispatch 任务队列

本目录用于跨窗口转交任务。它把“某窗口口头建议转交”升级为“可追踪、可接收、可执行的任务队列”。

## 定位

- `docs/tasks`：每个窗口长期要做什么，偏窗口级安排。
- `docs/dispatch`：某个具体转交任务实例，偏跨窗口流转。
- `docs/topics`：任务完成后的研究记录和结论沉淀。

## 目录结构

```text
docs/dispatch/
  README.md
  queue/
    T-YYYYMMDD-001-<slug>.md
  archive/
    README.md
```

## 任务状态

- `Inbox`：任务已提出，但目标窗口未确认。
- `Ready`：任务已分配，等待目标窗口接收。
- `Active`：目标窗口正在处理。
- `Blocked`：缺少资料、环境或决策。
- `Done`：目标窗口已完成，等待总控归档。
- `Dropped`：用户明确放弃，不再处理。
- `Archived`：已归档，不再出现在接单队列中。

## 优先级

- `P0`：阻塞整个研究体系。
- `P1`：影响当前主要流程。
- `P2`：常规重要任务。
- `P3`：可选优化或低优先级实验。

## 统一命令

窗口收到新需求前，先做闭环检查：

```text
检查窗口：软件开发
```

更明确的写法：

```text
检查Dispatch：软件开发
```

查看某窗口可接任务：

```text
接单：软件开发
```

更明确的写法：

```text
接Dispatch：软件开发
```

查看队列但不执行：

```text
查看队列：软件开发
```

执行指定任务：

```text
执行任务：T-20260509-001
```

## 接单规则

## 窗口新指令前置检查

用户是健忘的。窗口收到新需求时，应先做轻量闭环检查：

1. 检查 Outbox：本窗口作为 `Source Window` 转出去、但状态不是 `Done`、`Dropped`、`Archived` 的任务。
2. 检查 Inbox：本窗口作为 `Target Window`，状态为 `Inbox` 或 `Ready` 的任务。
3. 如果没有未闭环任务，继续处理用户的新需求。
4. 如果有未闭环任务，先简短提示用户，并给出建议：
   - 先处理某个更高优先级队列任务。
   - 继续处理当前新需求。
   - 放弃或延后某个队列任务。

窗口不应因为存在队列任务就强行中断用户的新需求；它应把选择权交给用户。

建议回复格式：

```text
我先检查到：

待接收任务：
- T-...：...

本窗口转出但未闭环：
- T-...：...

建议：
1. 先处理 ...
2. 继续你刚才的新需求
3. 将 ... 标记为 Dropped 或保持 Ready
```

## 接单规则

窗口收到 `检查窗口：<窗口名>` 或 `检查Dispatch：<窗口名>` 后，只做 Inbox 和 Outbox 检查，不执行任务。

窗口收到 `接单：<窗口名>` 后：

1. 读取 `AGENTS.md`。
2. 读取 `docs/dispatch/README.md`。
3. 读取自己的 `docs/tasks/<window>.md` 和 `docs/windows/<window>.md`。
4. 在 `docs/dispatch/queue` 中筛选 `Target Window` 等于当前窗口，且 `Status` 为 `Ready` 或 `Inbox` 的任务。
5. 按 `Priority` 从高到低、再按 `Created` 从早到晚排序。
6. 选择第一个任务，确认是否属于自身边界。
7. 如果属于自身边界，开始处理。
8. 如果不属于自身边界，把任务状态改为 `Blocked` 或返回总控重新分派。

`接单：<窗口名>` 只处理 `docs/dispatch/queue` 中的任务，不处理 `docs/tasks/<window>.md` 里的窗口当前任务。

如果用户要继续窗口当前任务，应使用：

```text
继续当前窗口任务
```

## 任务完成后的要求

目标窗口完成任务后，应：

- 更新任务文件状态为 `Done`。
- 在任务文件中写入结果摘要和验证方式。
- 回填到对应 `docs/topics/*.md`。
- 通知总控归档或更新索引。

## 用户选择处理策略

当用户在正确窗口提出新需求，且该窗口同时存在待接收任务：

- 如果新需求和待接收任务相关，建议合并处理。
- 如果新需求优先级更高，先处理新需求，并保持队列任务为 `Ready`。
- 如果用户明确放弃队列任务，把状态改为 `Dropped`，并写入原因。
- 如果用户只是暂时不处理，保持 `Ready` 或改为 `Blocked`，不要删除任务。

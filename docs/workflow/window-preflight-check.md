# 窗口新指令前置检查

本流程用于解决用户在错误窗口提出需求后，转交任务可能被遗忘的问题。

## 目标

每个窗口收到新需求时，先检查：

- 本窗口转出去的任务是否已经完成。
- 其他窗口转给本窗口的任务是否还没处理。
- 当前新需求是否应优先于已有队列任务。

## 检查类型

### Inbox

本窗口是 `Target Window`，且任务状态为 `Inbox` 或 `Ready`。

含义：别人转给本窗口、等待本窗口接收。

### Outbox

本窗口是 `Source Window`，且任务状态不是 `Done`、`Dropped`、`Archived`。

含义：本窗口曾经转出任务，但还没有闭环。

## 建议决策

窗口检查后，不应强制用户处理旧任务，而应给出建议：

1. 如果有 P0/P1 任务，建议先处理。
2. 如果新需求和已有任务相关，建议合并处理。
3. 如果新需求更紧急，建议保持已有任务为 `Ready`，先处理新需求。
4. 如果用户明确不再需要旧任务，建议标记为 `Dropped`。

## 输出规则

前置检查是常规安全动作，不应每次都用固定话术展示。

默认行为：

- 没有待处理 Inbox/Outbox、没有边界冲突时，静默完成检查，正常自然回复。
- 发现待处理任务、边界冲突、转交选择、文件修改、Git/发布动作、隐私风险或上下文混淆时，才明确说明检查结果。
- 用户明确发送 `检查窗口`、`检查Dispatch`、`查看队列` 或类似命令时，应完整展示检查结果。

用户可见的实时进度更新也遵守同一规则：

- 不要在进度更新中声明“正在做常规 dispatch 预检”“先按窗口规则检查”等无异常流程。
- 进度更新只说明具体正在做的工作、发现的问题、即将修改的文件、验证动作、Git/发布动作或需要用户取舍的决策。
- 如果检查没有发现异常，最终回复和中间进度都不需要提及这次检查。

## 推荐回复格式

下面格式只用于发现问题、用户明确要求检查，或需要用户做取舍时。普通对话不要机械套用。

```text
我先做了窗口前置检查。

Inbox：
- T-...：...

Outbox：
- T-...：...

这次新需求：
- ...

建议：
1. 先处理 ...
2. 或继续当前新需求
3. 对未处理任务保持 Ready / 标记 Dropped / 改为 Blocked
```

## 适用命令

```text
检查窗口：软件开发
```

更明确的命令：

```text
检查Dispatch：软件开发
```

```text
查看队列：软件开发
```

```text
接Dispatch：软件开发
```

## 验证记录

### 2026-05-09 写作剧本窗口漏检

用户在 `写作剧本` 窗口发送：

```text
检查窗口：写作剧本
```

窗口只检查了 `docs/tasks/writing-screenplay.md` 和相关工作指南，返回“没有发现 Ready 待处理任务”，但本地 Dispatch 队列中实际存在：

```text
T-20260509-001 [Ready] Target Window: 写作剧本
```

判断：`检查窗口` 命令仍有歧义，容易被解释为检查窗口本身状态，而不是检查 `docs/dispatch/queue`。

调整：

- 新增更明确命令 `检查Dispatch：<窗口名>`。
- 明确要求 `检查窗口` 和 `检查Dispatch` 都必须读取 `docs/dispatch/queue/*.md`。
- 后续优先使用 `检查Dispatch：写作剧本` 复测。
